Primer Parcial: SGBD, Subconsultas, Índices, Hashing y Optimización de Consultas

1️⃣ Sistema Gestor de Base de Datos (SGBD)

¿Qué es un SGBD?
Es el software que actúa como interfaz entre la base de datos, los usuarios y las aplicaciones.
Funciones Principales ›
  • Administrar, almacenar y recuperar información de forma eficiente
  • Garantizar la fiabilidad de grandes cantidades de datos
  • Permitir operaciones CRUD: Crear, Leer, Actualizar, Eliminar
  • Asegurar integridad, consistencia y seguridad de los datos

Ejemplos: Oracle, SQL Server, PostgreSQL, MySQL, MongoDB, Redis

Problemas ANTES de SGBD ›
  • Redundancia e inconsistencia: Los datos se duplicaban y no coincidían entre archivos
  • Dificultad para acceder: Información dispersa, lenta de recuperar
  • Aislamiento: Datos en formatos y ubicaciones diferentes
  • Falta de integridad: Difícil aplicar reglas de validación
  • Sin atomicidad: No se garantizaba completitud de operaciones
  • Acceso concurrente: Anomalías con múltiples usuarios
  • Seguridad deficiente: Imposible restringir acceso con precisión
Niveles de Abstracción de Datos
Los niveles de abstracción son distintas formas de ver una base de datos según quién la use, ocultando los detalles innecesarios.
Propósito: Sirven para separar: cómo se guardan los datos, cómo están organizados, qué parte ve cada usuario.
Nivel 1: Físico (Bajo) ›

Describe cómo se almacenan físicamente los datos en el disco.

  • Estructuras de archivos
  • Índices y bloques
  • Asignación de espacio
  • Ubicación exacta en disco
Nivel 2: Lógico (Conceptual) ›

Describe QUÉ datos se almacenan y qué RELACIONES existen entre ellos, ocultando detalles físicos.

  • Tablas y sus campos
  • Relaciones entre tablas
  • Atributos y tipos de datos
  • Restricciones de integridad
Nivel 3: Vistas (Externo) ›

Muestra solo una PARTE de la base de datos a usuarios específicos para simplificar su interacción.

  • Cajero: Ve saldos y operaciones bancarias
  • Cliente: Ve solo su propia cuenta
  • Gerente: Puede ver TODA la información

Cada usuario tiene una "vista" personalizada según su rol y permisos.

Los tres niveles trabajan juntos: el físico almacena, el lógico organiza y el externo presenta información personalizada.
Tipos de Usuarios del SGBD
Usuarios Normales (No Sofisticados) ›

No tienen conocimiento técnico profundo.

  • Interactúan mediante programas de aplicación previamente escritos
  • No escriben código ni consultas SQL
  • Usan interfaces gráficas amigables
  • Ejemplo: Cajero usando software bancario, vendedor usando POS
Programadores de Aplicaciones ›

Profesionales informáticos que crean el software.

  • Escriben programas y desarrollan interfaces
  • Conocen lenguajes de programación (Java, C#, Python, etc.)
  • Conectan aplicaciones con la base de datos
  • Manejan APIs y consultas SQL desde código
Usuarios Sofisticados ›

Interactúan directamente sin escribir programas.

  • Formulan consultas usando lenguajes de consulta (SQL, QBE)
  • Remiten consultas directamente al procesador de consultas
  • No necesitan intermediario de programador
  • Ejemplo: Analista de datos, investigador, ejecutivo consultando reportes
El Administrador de Bases de Datos (DBA)
Es la persona responsable de gestionar de manera centralizada los datos y los programas que los acceden. Es el "guardián" de la base de datos.
Funciones del DBA ›

Funciones estratégicas:

  • Definir el esquema: Estructura y organización de la base de datos
  • Establecer estructura de acceso: Crear índices, optimizar consultas
  • Autorizar permisos: Quién puede ver/modificar qué datos

Mantenimiento rutinario:

  • Realizar copias de seguridad (backups): Proteger contra pérdida de datos
  • Gestionar espacio en disco: Monitorear y asignar almacenamiento
  • Ajustar rendimiento (tuning): Optimizar velocidad y eficiencia
  • Monitoreo: Vigilar el funcionamiento del SGBD
Responsabilidades Clave ›
Seguridad de datos
Integridad de BD
Disponibilidad 24/7
Recuperación ante fallos
Optimización de performance
Gestión de usuarios

El DBA debe garantizar: Que los datos sean seguros, accesibles, consistentes y rápidos de acceder.

⚠️ Nota: El DBA es una posición crítica. Sin un buen DBA, la base de datos se vuelve lenta, insegura y propensa a pérdida de datos.

2️⃣ Subconsultas

Es una consulta anidada dentro de otra sentencia SQL. Se encierran entre paréntesis.
Subconsultas Escalares ›
  • Se usan con operadores: =, >, <, >=, <=, !=
  • Devuelven: Un solo valor
  • Ejemplo: Encontrar el producto con la cantidad máxima
  • Error común: Si devuelve más de un valor cuando se espera uno solo → ERROR
Subconsultas de Lista ›
  • Se utilizan con: Cláusula IN
  • Devuelven: Múltiples valores (lista)
  • Ejemplo: Buscar productos cuya cantidad sea mayor a 100
Subconsultas Correlacionadas ›
Funcionamiento: Las subconsultas correlacionadas dependen de los valores de la consulta externa: para cada fila de la consulta externa, se ejecuta la subconsulta interna, pasando un valor de la externa a la interna, y se repite hasta evaluar todas las filas.
Modificación de Datos con Subconsultas
Podés usar subconsultas para realizar cambios masivos basados en otras tablas. Es una forma poderosa de actualizar múltiples registros según condiciones complejas.
Uso común: Sincronizar datos entre tablas, actualizar valores según cálculos de otras tablas, copiar información filtrada.
INSERT ... SELECT ›

Inserta filas en una tabla destino basándose en el resultado de un SELECT de otra tabla.

Sintaxis básica:

INSERT INTO TablaDestino (col1, col2, col3) SELECT col1, col2, col3 FROM TablaOrigen WHERE condición

Ejemplo práctico:

-- Copiar empleados de Argentina a una tabla de Customers INSERT INTO Customers (Name, Country, Email) SELECT EmployeeName, Country, EmployeeEmail FROM Employees WHERE Country = 'Argentina' AND Active = 1

Consideraciones importantes:

  • Tipos de datos: Las columnas del SELECT deben coincidir con tipos y cantidad de la tabla destino
  • Restricciones: Si hay claves primarias duplicadas, la inserción falla
  • Orden: Asegúrate de especificar las columnas en el mismo orden
  • Omitir columnas: Si una columna no aparece, toma el valor por defecto (DEFAULT) o NULL
  • Transacción: Considerar usar BEGIN TRANSACTION para revertir si hay error
DELETE con Subconsultas ›

Elimina filas según condiciones complejas definidas en otra tabla.

Opción 1: Usar WHERE ... IN con subconsulta

DELETE FROM Pedidos WHERE ClienteID IN ( SELECT ClienteID FROM Clientes WHERE País = 'España' AND Estado = 'Inactivo' )

Elimina todos los pedidos de clientes españoles inactivos

Opción 2: Usar JOIN en FROM

DELETE FROM Pedidos FROM Pedidos p INNER JOIN Clientes c ON p.ClienteID = c.ClienteID WHERE c.País = 'España' AND c.Estado = 'Inactivo'

Mismo resultado pero usando JOIN (más legible en SQL Server)

Cuidados críticos:

  • ⚠️ Siempre verificar la subconsulta primero: Ejecuta el SELECT aparte para confirmar qué borrará
  • Usar transacciones: BEGIN TRANSACTION + ROLLBACK si es necesario
  • Respetar restricciones: Si hay claves foráneas, algunos borrados pueden estar prohibidos
  • Performance: Borrar muchos registros puede ser lento
🔴 PELIGRO: Un DELETE sin WHERE elimina TODA la tabla. Siempre especifica condiciones y prueba primero.
UPDATE con Subconsultas ›

Modifica valores usando datos de otra tabla como referencia.

Opción 1: Usar subconsulta en WHERE (condicionar qué filas actualizar)

-- Aumentar precios solo a productos de proveedores españoles UPDATE Productos SET Precio = Precio * 1.15 -- Aumenta 15% WHERE ProveedorID IN ( SELECT ProveedorID FROM Proveedores WHERE País = 'España' )

Aumenta precios solo de productos del proveedor especificado

Opción 2: Usar SET con subconsulta (actualizar valor calculado)

-- Actualizar stock de producto con el total de sus ventas UPDATE Productos SET StockVendido = ( SELECT SUM(Cantidad) FROM Pedidos WHERE Pedidos.ProductoID = Productos.ProductoID ) WHERE ProductoID > 0

Calcula suma desde otra tabla e inserta el resultado

Opción 3: Usar JOIN (actualizar basándose en relaciones)

-- Actualizar el nombre del proveedor en la tabla de Productos UPDATE Productos SET NombreProveedor = p.Nombre FROM Productos pr INNER JOIN Proveedores p ON pr.ProveedorID = p.ProveedorID WHERE pr.Precio > 100

Sincroniza información entre tablas usando relaciones

Ejemplo avanzado: Actualizar múltiples columnas

-- Actualizar precio y categoría según datos del proveedor UPDATE Productos SET Precio = Precio * 1.10, Categoría = (SELECT Especialidad FROM Proveedores WHERE ProveedorID = Productos.ProveedorID) WHERE ProveedorID IN ( SELECT ProveedorID FROM Proveedores WHERE País = 'Brasil' AND Activo = 1 )

Mejores prácticas en UPDATE:

  • Verificar primero: Ejecuta un SELECT con las mismas condiciones para ver qué va a cambiar
  • Usar transacciones: BEGIN / COMMIT / ROLLBACK
  • Limitar filas: Usar WHERE específico, no actualizar TODA la tabla innecesariamente
  • Subconsultas correlacionadas: Pueden ser lentas si hay muchos registros
  • Cálculos complejos: Si la subconsulta devuelve NULL, el UPDATE pone NULL (salvo que uses ISNULL)
Truco: Usa ISNULL(subconsulta, valor_por_defecto) para manejar valores NULL en updates
Tabla Comparativa: INSERT vs DELETE vs UPDATE ›
Operación Propósito Sintaxis Clave Riesgo
INSERT...SELECT Copiar datos filtrados de otra tabla SELECT + WHERE Duplicados si claves ya existen
DELETE Eliminar filas permanentemente IN (subconsulta) o JOIN ⚠️ Pérdida irreversible de datos
UPDATE Modificar valores existentes SET + WHERE/JOIN/subconsulta Cambios inesperados, inconsistencias

3️⃣ Índices

¿Qué es un Índice? ›

Es una estructura de datos diseñada específicamente para mejorar la velocidad de acceso a los registros de una tabla.

[Clave de búsqueda | Puntero]

Ventajas:

  • Archivo de índice es MUCHO más pequeño que el archivo de datos original
  • Procesamiento más rápido
  • Acceso directo sin búsquedas lineales
Clases Básicas de Índices ›
Tipo de Índice ¿Cómo funciona?
Índices Ordenados Las claves se almacenan en orden alfabético o numérico
Índices de Hash Las claves se distribuyen uniformemente en cajones (buckets) mediante función matemática
Tiempos de Acceso ›

Resumen técnico (b = cantidad de bloques de memoria):

Tipo de Almacenamiento Fórmula Descripción
Heap (Desordenado) b/2 promedio Búsqueda lineal, muy lento
Ordenado (Binaria) log₂(b) promedio Mucho más eficiente
Índice Primario (Clustered) ›

La clave coincide con el orden físico en el que están guardados los registros en disco.

  • Cantidad: Solo UNO por tabla
  • Eficiencia: Muy bueno para búsquedas de rango
  • Relación con clave primaria: Usualmente se basan en ella
Índice Secundario (Non-Clustered) ›

La clave especifica un orden distinto al orden físico.

  • Cantidad: Puede haber MUCHOS por tabla
  • Densidad: SIEMPRE deben ser densos
  • Eficiencia: Útil para búsquedas puntuales
Árbol B+ (Estructura Moderna) ›

Estructura jerárquica para búsqueda extremadamente rápida.

  • Equilibrio perfecto: Todos los caminos miden igual
  • Auto-reorganización: Se ajusta automáticamente
  • Dinámico: No requiere reorganizar todo periódicamente
  • Eficiente: Óptimo para búsquedas, inserciones y eliminaciones
Índices Agrupados (Clustering Index) ›

Definición: Es aquel índice cuya clave de búsqueda coincide con el orden físico (secuencial) en el que están guardados los registros en el archivo.

Sinonimia: En el contexto de la materia, a estos índices también se los denomina índices primarios.

Estructura: Aparece un registro en el índice por cada valor de la clave de búsqueda que existe en el archivo de datos.

Relación con los datos: Establece una correspondencia directa entre la organización del índice y cómo están físicamente distribuidos los datos en el disco.

Características clave: Solo UNO por tabla, muy eficiente para rangos, orden lógico = orden físico.
Índice Denso vs Índice Disperso ›
Índice Denso (Dense Index) Índice Disperso (Sparse Index)
Existe una entrada de índice por cada registro del archivo de datos. Existe una entrada de índice por cada bloque de datos, no por cada registro.
Permite acceder directamente al registro buscado. Primero localiza el bloque y luego realiza una búsqueda secuencial dentro de ese bloque.
Ocupa más espacio porque contiene muchas más entradas. Ocupa mucho menos espacio al almacenar menos entradas.
Se utiliza principalmente en índices secundarios (Non-Clustered). Se utiliza principalmente en índices primarios o agrupados (Clustered).

📌 Funcionamiento del Índice Disperso

  • Almacenamiento: Guarda solo una entrada por cada bloque de datos.
  • Búsqueda: Localiza en el índice la mayor clave menor o igual a la buscada.
  • Recorrido: Accede al bloque indicado y busca secuencialmente el registro.
Recordá: El índice denso tiene una entrada por registro, mientras que el índice disperso tiene una entrada por bloque. Esa es la diferencia más importante para el examen.
Ventajas de Eficiencia ›
  • Menos espacio: El archivo de índice ocupa muchos menos bloques que el archivo de datos.
  • Menos entradas: Al haber menos registros en el índice, es más liviano y rápido de procesar.
  • Simplicidad: Cada entrada contiene únicamente Clave + Puntero.
Actualización del Índice (Mantenimiento) ›

🟢 Inserción

Índice Denso Índice Disperso
• Si la clave no existe → se crea una nueva entrada.

• Si la clave existe y guarda todos los punteros → se agrega el nuevo puntero.

• Si solo apunta al primero → el nuevo registro se inserta físicamente después de los registros con esa clave. 		Solo se actualiza cuando el nuevo registro pasa a ser el menor valor del bloque.

🔴 Borrado

Índice Denso Índice Disperso
• Se elimina el puntero correspondiente.

• Si era el primer registro con esa clave, el índice pasa a apuntar al siguiente. 		• Si la clave no estaba indexada → no se modifica.

• Si era la única entrada → se reemplaza por la siguiente clave disponible.

• Si apuntaba al registro eliminado → se actualiza al siguiente registro con la misma clave.
Índices Multinivel ›

Se utilizan cuando el índice primario es demasiado grande para cargarse completamente en memoria RAM.

Concepto:

Se trata al índice primario como si fuera un archivo secuencial y se construye un índice disperso por encima de él.

Componente Descripción
Índice Interno Es el índice primario original.
Índice Externo Índice disperso que apunta al índice interno.
  • Escalabilidad: Si el índice externo también es muy grande, se puede crear otro nivel superior.
  • Propósito: Reducir la cantidad de accesos al disco y acelerar la búsqueda.
Idea clave: Es un índice sobre otro índice.

4️⃣ Hashing (Asociación)

Técnica para organizar archivos que permite acceder a los datos DIRECTAMENTE, sin recorrer índices jerárquicos ni realizar búsquedas binarias, reduciendo la cantidad de operaciones de Entrada/Salida (E/S).
¿Qué es la Asociación (Hashing)? ›

Es una técnica de organización de archivos que permite acceder a un registro de forma directa, sin necesidad de recorrer un índice jerárquico.

Concepto Definición
Cajón (Bucket) Unidad de almacenamiento (normalmente un bloque de disco) donde se guardan uno o más registros.
Función de Asociación (h) Función matemática que recibe una clave de búsqueda y devuelve la dirección del cajón donde se almacena el registro.
h(Ki) = Dirección del Cajón
Idea clave: Hashing busca acceder directamente al dato sin recorrer un árbol o realizar búsquedas binarias.
Asociación Estática (Hashing Estático) ›

La Asociación Estática utiliza un conjunto fijo de cajones. La función h(K) siempre asigna las claves a ese mismo conjunto de direcciones.

  • Número de cajones: Se define al crear el archivo y no cambia.
  • Función h: Mapea cada clave al mismo conjunto fijo de direcciones.
  • Problema principal: Cuando muchas claves llegan al mismo cajón aparecen las colisiones y los desbordamientos.
Recordá: En Hashing Estático el tamaño de la estructura NO cambia aunque aumente la cantidad de registros.
Operaciones Principales ›
Operación ¿Cómo funciona?
Búsqueda Se calcula h(Ki) para obtener el cajón correspondiente. Si existen colisiones, deben revisarse todos los registros del cajón hasta encontrar el correcto.
Inserción Se calcula h(Ki) y el nuevo registro se almacena en el cajón obtenido.
Borrado Se localiza el cajón mediante h(Ki), se encuentra el registro y se elimina.
Todas las operaciones comienzan calculando la función h(K).
Propósitos de la Asociación ›
Tipo Descripción
Organización de Archivo Asociativo Permite obtener directamente la dirección del bloque de disco donde se encuentra el registro buscado.
Índice Asociativo (Hash Index) Organiza las claves de búsqueda junto con sus punteros. Siempre es un índice secundario.
Funciones de Asociación y Distribución ›

El objetivo de una función de asociación es distribuir las claves de la forma más eficiente posible, minimizando las colisiones y aprovechando correctamente el espacio disponible.

Característica Descripción
Distribución Uniforme Cada cajón recibe aproximadamente la misma cantidad de claves.
Distribución Aleatoria Los valores obtenidos no deben seguir un orden externo para evitar agrupamientos.
Objetivo Reducir las colisiones y aprovechar al máximo el espacio disponible.
La peor función posible: aquella que asigna todas las claves al mismo cajón.
Colisiones ›

Una colisión ocurre cuando dos o más claves diferentes producen el mismo valor de la función de asociación.

h(K5) = h(K7)

En este caso, ambas claves intentan almacenarse en el mismo cajón.

Cuando existen colisiones es necesario aplicar un método para resolverlas.
Desbordamientos y Atascos ›

El desbordamiento ocurre cuando un cajón ya no tiene espacio para almacenar nuevos registros.

Las causas más comunes son:

  • Mala distribución de la función hash.
  • Demasiadas colisiones.
  • Crecimiento de la base de datos.
El desbordamiento disminuye considerablemente el rendimiento del acceso directo.
Resolución de Desbordamientos ›
Asociación Cerrada Asociación Abierta
Utiliza cajones de desbordamiento adicionales conectados mediante una lista enlazada (cadena de desbordamiento). No utiliza bloques extra. Si el cajón está lleno, busca otro cajón libre, normalmente el siguiente en forma cíclica.
✔ Más sencilla para mantener los datos relacionados. ✔ Aprovecha únicamente los cajones originales.
Recordá: Cerrada = usa cajones extra.
Abierta = busca otro cajón disponible.
Estrategias ante el Crecimiento de Datos ›

Como las bases de datos son dinámicas, el Hashing Estático debe prever cómo manejar el crecimiento de registros.

Estrategia Ventaja Desventaja
Tamaño Actual Se optimiza para la cantidad actual de registros. El rendimiento disminuye cuando la base crece.
Tamaño Previsto Evita reorganizaciones futuras. Desperdicia mucho espacio al comienzo.
Reorganización Periódica Permite reajustar completamente la estructura. Es una operación lenta que recalcula todas las funciones hash y requiere bloquear el acceso al archivo.
Durante una reorganización periódica normalmente se debe impedir el acceso al archivo hasta finalizar el proceso.
Asociación Dinámica (Hashing Extensible) ›

A diferencia del Hashing Estático, el Hashing Dinámico permite modificar la función de asociación y la cantidad de cajones para adaptarse automáticamente al crecimiento o disminución de la base de datos.

Característica Descripción
Adaptabilidad Divide o fusiona cajones según el tamaño de la base de datos.
Bajo impacto La reorganización afecta únicamente a un cajón por vez.
Cajones bajo demanda Los cajones se crean únicamente cuando son necesarios.
Ventaja Mantiene la eficiencia espacial y evita reorganizaciones masivas.
Idea clave: El Hashing Dinámico crece junto con la base de datos.
Aspectos Técnicos ›
Elemento Descripción
Función h Genera valores en un rango amplio, normalmente enteros binarios de 32 bits.
Bits utilizados (i) Se utilizan los primeros i bits como prefijo asociativo.
Valor de i Puede aumentar o disminuir según crece o disminuye la base de datos.
Tabla de direcciones Los bits obtenidos actúan como desplazamiento dentro de la tabla de direcciones de cajones.
0 ≤ i ≤ b     (b = 32 bits normalmente)
Índices Ordenados vs Asociación (Hashing) ›

📍 Consultas de Igualdad

WHERE Ai = c
Índices Ordenados Hashing
El tiempo de búsqueda depende del tamaño de la base de datos. El tiempo medio es prácticamente constante, independientemente del tamaño de la base.
Ganador: Hashing.

📍 Consultas por Rango

WHERE Ai >= c1 AND Ai <= c2
Índices Ordenados Hashing
Localiza el primer valor y continúa recorriendo los siguientes registros en orden. No existe un "siguiente cajón", ya que las claves se distribuyen de forma aleatoria.
Ganador: Índices Ordenados (Árbol B+).
Accesos Multiclave ›

Cuando una consulta posee varias condiciones (por ejemplo, sucursal y saldo), existen tres estrategias para acceder a los datos.

Estrategia Descripción
Índice Simple Se utiliza un solo índice para una condición y luego se filtran manualmente los resultados para cumplir la segunda condición.
Intersección de Punteros Se utilizan dos índices independientes. Luego se intersectan ambas listas de punteros y solo permanecen los registros presentes en ambas.
Clave de Búsqueda Compuesta Se crea un único índice utilizando la concatenación de varios atributos (por ejemplo: NombreSucursal + Saldo).
Recordá:
  • Índice Simple: filtra después.
  • Intersección: combina punteros de dos índices.
  • Clave Compuesta: crea un único índice con todos los atributos.

5️⃣ Optimización de Consultas

Proceso mediante el cual el SGBD analiza una consulta SQL, la optimiza y ejecuta utilizando el plan de menor costo posible.
Procesamiento de Consultas ›

Objetivo: Obtener el plan de ejecución más eficiente.

Componente Función
Usuario Envía la consulta SQL.
Procesador Interpreta la consulta.
Optimizador Selecciona el mejor plan.
Motor de almacenamiento Accede a los datos.
Ejecutor Ejecuta el plan.
Tipos de Optimizador ›
Basado en Sintaxis Basado en Costos
Heurístico. Evalúa distintos planes.
Depende del SQL escrito. Usa estadísticas.
Siempre genera el mismo plan. Selecciona el plan de menor costo.
No analiza índices. Analiza índices, cardinalidad y selectividad.
Recordá: Sintaxis = cómo está escrito el SQL. Costos = cuál plan cuesta menos.
Etapas del Procesamiento ›

Evaluación de Expresiones y Heurísticas ›

Evaluación: El SGBD construye un árbol de operadores y decide:

  • Algoritmos a utilizar.
  • Orden de ejecución.
  • Plan de menor costo.

Tipos de optimización:

Basada en Costos Heurística
Calcula el costo de cada plan. Aplica reglas generales de optimización.

Reglas Heurísticas

  • Selección temprana: filtrar registros lo antes posible.
  • Proyección temprana: reducir columnas cuanto antes.
  • Evitar productos cartesianos: son muy costosos.
Para recordar: Seleccionar temprano, proyectar temprano y evitar productos cartesianos.
