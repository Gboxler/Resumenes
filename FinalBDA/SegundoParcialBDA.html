Segundo Parcial: Transacciones, Concurrencia y Seguridad Informática

1️⃣ Transacciones

Una transacción es un conjunto de operaciones relacionadas que se ejecutan como una sola unidad de trabajo. Debe ejecutarse completamente o no ejecutarse en absoluto.
Ejemplo: Compra de un Producto ›
  • Se descuenta una unidad del stock.
  • Se registra la venta.

Si alguna de las dos operaciones falla, todos los cambios deben deshacerse para evitar inconsistencias.

Idea clave: Una transacción funciona con la regla "todo o nada". Si una operación falla, deben deshacerse todos los cambios realizados.
Propiedades ACID ›

Atomicidad (Atomicity): Todo o nada. Si una operación falla, todas las operaciones realizadas deben revertirse.

Consistencia (Consistency): La base de datos debe pasar de un estado válido a otro estado válido, respetando todas las reglas de integridad.

Aislamiento (Isolation): Las transacciones que se ejecutan al mismo tiempo no deben interferir entre sí.

Durabilidad (Durability): Una vez confirmado un cambio mediante COMMIT, este permanece incluso ante fallas del sistema.

Estados de una Transacción ›

Las transacciones pueden encontrarse en diferentes estados según el momento de su ejecución.

  • Activa: Estado inicial mientras la transacción se está ejecutando.
  • Parcialmente comprometida: Se ejecutó la última instrucción, pero aún no se confirmó definitivamente mediante COMMIT.
  • Fallida: La transacción detectó un error y no puede continuar.
  • Abortada o Retrocedida: Se ejecuta un ROLLBACK, se deshacen los cambios realizados y la base de datos vuelve al estado anterior al inicio de la transacción.
  • Comprometida: La transacción finaliza correctamente y los cambios quedan guardados mediante COMMIT.

Luego de Abortada, la transacción puede:

  • Reiniciarse para intentar ejecutarse nuevamente.
  • Cancelarse definitivamente.
Control de Transacciones ›
Tipo Descripción
Implícitas El sistema las inicia y finaliza automáticamente. Ejemplos: INSERT, UPDATE, DELETE
Explícitas El programador controla manualmente la transacción mediante BEGIN TRANSACTION, COMMIT, ROLLBACK

Comandos principales:

  • ✔ BEGIN TRANSACTION: Inicia una transacción.
  • ✔ COMMIT: Confirma los cambios realizados.
  • ✔ ROLLBACK: Deshace todos los cambios realizados desde el inicio de la transacción.
Buenas Prácticas ›
  • Mantener las transacciones cortas.
  • Dividir transacciones muy extensas.
  • Utilizar índices adecuados para mejorar el rendimiento.
  • Evitar transacciones anidadas cuando sea posible.
Algoritmo Deshacer/Rehacer (Undo/Redo) ›

Ante una falla, el sistema utiliza el Log para recuperar la base de datos.

Operación Descripción
Deshacer (UNDO / Roll Back) Deshace las transacciones que quedaron incompletas o no fueron confirmadas.
Rehacer (REDO / Roll Forward) Vuelve a aplicar los cambios de las transacciones que ya habían sido confirmadas.
Recuperación de Medio: Ante fallas de hardware (por ejemplo, rotura de disco), se utiliza un Backup y el Log para restaurar la información.

2️⃣ Concurrencia

La concurrencia permite que varias transacciones se ejecuten al mismo tiempo. Es fundamental en sistemas multiusuario donde varias personas acceden simultáneamente a la base de datos.
Objetivos de la Concurrencia ›
  • Mejorar el rendimiento.
  • Aprovechar mejor los recursos.
  • Incrementar la productividad.
  • Reducir tiempos de espera.
Problema principal: Cuando varias transacciones trabajan sobre los mismos datos al mismo tiempo, pueden producirse inconsistencias.
Concurrencia y Secuencialidad ›

La concurrencia permite que varias transacciones se ejecuten al mismo tiempo. Sin embargo, para mantener la consistencia de la base de datos, el resultado obtenido debe ser el mismo que si las transacciones se hubieran ejecutado de forma secuencial, es decir, una detrás de la otra.

Para verificar esta equivalencia existen dos criterios:

  • Secuencialidad por conflictos: se basa en analizar los conflictos entre transacciones. Existe un conflicto cuando dos transacciones acceden al mismo dato y al menos una de ellas lo modifica.
  • Secuencialidad por vistas: se cumple cuando el resultado final observado por los usuarios es exactamente el mismo que se obtendría en una ejecución secuencial.
Idea clave: aunque las transacciones se ejecuten concurrentemente, el resultado final debe ser equivalente al de una ejecución secuencial.
Serialización ›

La serialización es el criterio que permite determinar si una ejecución concurrente es correcta.

Una ejecución es serializable cuando, aunque las transacciones se ejecuten al mismo tiempo, el resultado final es el mismo que se obtendría ejecutándolas una detrás de la otra (de forma secuencial).

Objetivo: mantener la consistencia de la base de datos y evitar resultados incorrectos producidos por la concurrencia. Generalmente, el control de la serialización es realizado por el planificador del SGBD.

Planificaciones Recuperables ›

Una planificación es recuperable cuando, si T2 leyó datos de T1, entonces T2 debe esperar el COMMIT de T1 antes de realizar su propio COMMIT. Si una transacción no depende de otra, puede confirmar normalmente.

Objetivo: Evitar que una transacción confirme resultados basados en datos que luego podrían ser anulados.

Ejemplo:

T1 escribe A = 200 T2 lee A = 200

✅ Recuperable: T2 espera el COMMIT de T1 antes de confirmar.

❌ No recuperable: T2 confirmó datos que provenían de T1, pero luego T1 canceló sus cambios (ROLLBACK).

Planificaciones Sin Cascada ›

Una planificación es sin cascada cuando una transacción solo puede leer datos que ya fueron confirmados mediante COMMIT.

Objetivo: Evitar retrocesos en cascada (Cascading Rollback) y lecturas sucias.

Ejemplo:

T1: escribe(A) → Todavía NO hizo COMMIT T2: leer(A) → ❌ No puede, debe esperar
Correcto (Sin Cascada): T2 solo puede leer A porque T1 ya confirmó ese dato.
Grafo de Precedencia ›

Es una herramienta utilizada para determinar si una planificación es serializable por conflictos.

Construcción:

  • Cada nodo representa una transacción.
  • Cada flecha (arista) representa un conflicto entre transacciones.

Regla:

✔ Si el grafo no tiene ciclos → la planificación es serializable.

❌ Si el grafo tiene ciclos → la planificación no es serializable.

Problemas de Concurrencia ›

Lectura Sucia (Dirty Read): Una transacción lee datos aún no confirmados por otra.

Lectura No Repetible: Ocurre cuando una transacción ejecuta dos veces la misma consulta y obtiene resultados diferentes porque otra transacción realizó un UPDATE o DELETE sobre esos datos y luego hizo COMMIT.

Lectura Fantasma (Phantom Read): Ocurre cuando una transacción ejecuta dos veces la misma consulta y obtiene una cantidad diferente de registros porque otra transacción agregó o eliminó registros y realizó COMMIT.

Actualización Perdida: Dos transacciones leen el mismo dato y ambas lo modifican, pero una sobrescribe los cambios de la otra, provocando que una actualización se pierda.

Control de Concurrencia: Bloqueos (Locks) ›

Son mecanismos que permiten controlar la concurrencia y garantizar la serialización. Para ello utilizan bloqueos.

Bloqueo Compartido (Shared Lock - C): Permite únicamente la lectura de los datos. Varias transacciones pueden leer el mismo dato al mismo tiempo, pero no modificarlo.

Bloqueo Exclusivo (Exclusive Lock - X): Permite leer y modificar los datos, pero solo una transacción puede hacerlo a la vez. Mientras una transacción tiene este bloqueo, ninguna otra puede acceder al dato.

Matriz de Compatibilidad:

Solicita Compartido (C) Exclusivo (X)
Compartido (C) ✔ Sí ✗ No
Exclusivo (X) ✗ No ✗ No
Estos mecanismos se utilizan para evitar inconsistencias en la base de datos y garantizar la serialización, asegurando que las transacciones no modifiquen simultáneamente los mismos datos.
Problemas con los Bloqueos ›

Inconsistencia por Tiempo de Desbloqueo: Se produce cuando una transacción libera un bloqueo demasiado pronto, permitiendo que otra transacción acceda a datos que todavía no representan un estado consistente.

Interbloqueo (Deadlock): Se produce cuando dos transacciones quedan esperando recursos bloqueados por la otra y ninguna puede continuar su ejecución. Para resolverlo, el sistema debe abortar o retroceder una de las transacciones, liberando sus bloqueos para que la otra pueda continuar.

Inanición (Starvation): Se produce cuando una transacción espera indefinidamente un recurso porque otras transacciones continúan obteniéndolo antes que ella.

Protocolos que Aseguran la Secuencialidad ›

Para garantizar que una ejecución concurrente produzca el mismo resultado que una ejecución secuencial, el SGBD utiliza distintos protocolos de control de concurrencia. Todos estos protocolos establecen reglas que indican cuándo una transacción puede bloquear, desbloquear, leer o escribir datos.

Tipos principales:

  • ✔ Protocolos basados en bloqueos (Locks)
  • ✔ Protocolos basados en grafos (Protocolo de Árbol)
  • ✔ Protocolos basados en marcas temporales (Timestamp)
Protocolo de Bloqueo en Dos Fases (2PL) ›

El protocolo más utilizado para garantizar la serialización. Consta de dos etapas:

Fase de crecimiento: La transacción obtiene los bloqueos que necesita. No puede liberar bloqueos.

Fase de decrecimiento: La transacción libera bloqueos. No puede obtener nuevos bloqueos.

Una vez que libera su primer bloqueo, no puede pedir otro.

Ventaja: Garantiza la serialización.

Desventajas: Puede producir interbloqueos (Deadlocks) y retrocesos en cascada.

Variantes:

Variante Descripción
PB2F Simple Fase de crecimiento (obtiene bloqueos) y fase de decrecimiento (libera bloqueos). Garantiza serialización, pero puede producir deadlocks y retrocesos en cascada.
PB2F Estricto Cumple las reglas del 2PL simple, pero además mantiene los bloqueos exclusivos (X) hasta que la transacción finaliza con COMMIT o ROLLBACK. Evita que otras transacciones lean datos modificados pero no confirmados, evitando los retrocesos en cascada.
PB2F Riguroso Más estricto: mantiene todos los bloqueos (compartidos y exclusivos) hasta que la transacción finaliza con COMMIT o ROLLBACK. Evita lecturas incorrectas y simplifica la recuperación, pero reduce la concurrencia.
PB2F Riguroso Refinado Mantiene las características del 2PL riguroso, pero permite conversiones de bloqueos: Upgrade (de C a X, solo en fase de crecimiento) y Downgrade (de X a C, solo en fase de decrecimiento). Aumenta la concurrencia.
Protocolos Basados en Grafos (Protocolo de Árbol) ›

Utilizan una estructura jerárquica (árbol) para determinar el orden en que se accede a los datos. Se emplean únicamente bloqueos exclusivos. El objetivo es asegurar la serialización evitando interbloqueos.

Reglas:

  • El primer bloqueo puede realizarse sobre cualquier dato.
  • Para bloquear un nodo hijo debe estar bloqueado previamente su nodo padre.
  • Los bloqueos pueden liberarse en cualquier momento.
  • Un dato desbloqueado no puede volver a bloquearse.
Ventajas Desventajas
✔ Garantiza la serialización por conflictos.
✔ Evita los interbloqueos.
✔ Permite liberar bloqueos antes que el protocolo 2PL.		 ✔ Puede generar bloqueos innecesarios.
✔ Reduce la concurrencia.

Ejemplo: si una transacción necesita acceder a J, puede verse obligada a bloquear también sus nodos padres.

Protocolos Basados en Marcas Temporales (Timestamp) ›

Asignan una marca temporal a cada transacción antes de comenzar su ejecución. La marca puede obtenerse mediante el reloj del sistema o un contador lógico. Cada transacción recibe una marca única.

Ejemplo:

MT(T1) = 10 MT(T2) = 20 Como 10 < 20, T1 tiene prioridad sobre T2.

Las operaciones deben ejecutarse respetando ese orden temporal.

Funcionamiento: para cada dato se almacenan dos marcas:

  • MT-E(Q): Última transacción que escribió el dato.
  • MT-L(Q): Última transacción que leyó el dato.

Estas marcas permiten decidir si una lectura o escritura es válida.

Ventajas Desventajas
✔ Garantizan la serialización.
✔ Evitan conflictos entre transacciones.
✔ No producen interbloqueos.		 ✔ Algunas transacciones pueden ser rechazadas y reiniciadas.
Granularidad Múltiple ›

Agrupa varios elementos de datos como una única unidad de sincronización o bloqueo. Permite aplicar bloqueos sobre distintos niveles de una estructura jerárquica.

Estructura jerárquica: Base de Datos (raíz) → Zonas → Archivos o Tablas → Registros o Filas. Cada nivel superior contiene a los niveles inferiores.

Funcionamiento: A diferencia del Protocolo de Árbol clásico, donde cada nodo representa un dato independiente, en la granularidad múltiple un nodo padre representa a todos sus nodos hijos. Cuando una transacción bloquea un nodo, el bloqueo se hereda automáticamente a todos sus descendientes.

Ejemplo: si una transacción bloquea una tabla, la tabla queda bloqueada y también quedan bloqueados todos sus registros.

Ventajas Desventajas
✔ Facilita la administración de bloqueos.
✔ Reduce la cantidad de bloqueos que deben gestionarse.		 ✔ Puede bloquear más datos de los necesarios.
✔ Puede disminuir la concurrencia.
Idea clave: en lugar de bloquear registro por registro, se puede bloquear un nivel superior (por ejemplo una tabla completa) y el bloqueo se aplica automáticamente a todos sus elementos hijos.
Niveles de Aislamiento (Estándar SQL-92) ›

Los niveles de aislamiento determinan cuánto puede afectar una transacción a otra.

Nivel Lectura Sucia Lectura No Repetible Lectura Fantasma
READ UNCOMMITTED ❌ Puede ocurrir ❌ Puede ocurrir ❌ Puede ocurrir
READ COMMITTED ✔ Evitada ❌ Puede ocurrir ❌ Puede ocurrir
REPEATABLE READ ✔ Evitada ✔ Evitada ❌ Puede ocurrir
SERIALIZABLE ✔ Evitada ✔ Evitada ✔ Evitada

READ UNCOMMITTED: Es el nivel más bajo, permite leer datos no confirmados.

READ COMMITTED: Solo permite leer datos confirmados.

REPEATABLE READ: Garantiza que una misma fila leída mantenga su valor durante toda la transacción.

SERIALIZABLE: Es el nivel más alto y más estricto, simula que las transacciones se ejecutan una después de otra.

3️⃣ Seguridad Informática

La seguridad informática busca proteger la información para garantizar que permanezca íntegra, disponible y accesible únicamente para usuarios autorizados. La información es uno de los activos más importantes de una organización.
¿Por qué proteger la información? ›

La información es un recurso estratégico y crítico para la organización. Su protección contribuye a:

  • Eficiencia de la gestión.
  • Ventajas competitivas.
  • Toma de decisiones.
  • Continuidad operativa.
¿Por qué es frágil la información? Porque depende de medios tecnológicos que pueden fallar y de personas que pueden dañarla o utilizarla indebidamente.
Impacto de la Informática ›

El uso de sistemas informáticos genera nuevos riesgos para la información:

  • Es posible realizar operaciones sin generar documentación fuente.
  • Es posible modificar información sin dejar huellas visibles.
  • La gran velocidad de procesamiento amplifica la capacidad humana y el impacto de los errores.
  • Los registros y pistas de auditoría son invisibles para el usuario común.
  • Es más fácil robar información cuando se encuentra concentrada en medios electrónicos.
  • La información electrónica es fácil de copiar, perder o destruir.
Idea clave: La informatización aumenta la eficiencia, pero también incrementa los riesgos de pérdida, modificación o robo de información, por lo que se requieren mayores medidas de seguridad.
Objetivos de Seguridad ›

✔ Integridad: La información solo puede ser modificada por usuarios autorizados.

✔ Disponibilidad: La información debe estar disponible cuando sea necesaria.

✔ Confidencialidad: La información solo puede ser conocida por personas autorizadas.

✔ Auditoría: Permite registrar y rastrear las actividades realizadas sobre la información.

Información a Proteger ›
✔ Digitalizada ✔ No Digitalizada
Servidores, computadoras, notebooks, discos, redes y comunicaciones. Documentos, planos, fórmulas, mapas, comunicaciones internas.
Marco Normativo ›

Se compone de:

  • Políticas.
  • Normas.
  • Procedimientos.
  • Estándares.
Factores de Riesgo ›

✔ Externos: Hackers, competidores, clientes, proveedores, traficantes de información.

✔ Internos: Empleados malintencionados, personal que busca beneficio propio, saboteadores.

Tipos de Seguridad ›

✔ Seguridad Física: Protege instalaciones y equipamiento.

Ejemplos: control de acceso físico, sistemas contra incendios, protección ante inundaciones.

✔ Seguridad Lógica: Protege sistemas y datos mediante usuarios, contraseñas, roles, permisos y perfiles.

Riesgos técnicos:

  • Intrusiones.
  • Virus.
  • Pérdida de datos.
  • Fallas de recuperación.

Controles de seguridad:

  • Contraseñas.
  • Perfiles de usuario.
  • Auditorías.
  • Backups.
  • Control de accesos físicos.
Desarrollo de un Plan de Seguridad ›
  1. Valuación de datos.
  2. Evaluación de riesgos.
  3. Determinación del alcance.
  4. Definición de medidas de seguridad.
  5. Implementación.
  6. Monitoreo y auditoría.
  7. Corrección y mejora.
Plan de Contingencia ›

Es el conjunto de procedimientos que permiten mantener la continuidad operativa ante incidentes o fallas.

Incluye:

  • Backups.
  • Recuperación de datos.
  • Equipamiento alternativo.
  • Procedimientos documentados.
  • Plan de recuperación por escrito.
  • Pruebas periódicas.

Eventos contemplados por un Plan de Contingencia:

✔ Naturales ✔ Accidentales ✔ Intencionales
Inundaciones, rayos, epidemias, sismos, terremotos, huracanes, nevadas. Incendios, cortes eléctricos, derrumbes, explosiones, interrupción de telecomunicaciones, filtraciones, emanaciones tóxicas. Robo o hurto, vandalismo, sabotaje, espionaje industrial, bombas o artefactos explosivos, incendios provocados.
¿Por qué Muchas Empresas No Se Preparan? ›
  • Falta de interés de la gerencia.
  • Presupuestos insuficientes.
  • Creencia de que "nunca va a pasar".
  • Dificultad para medir los beneficios de la seguridad.
Idea clave: La seguridad informática busca proteger la información garantizando Integridad, Disponibilidad, Confidencialidad y Auditoría, mediante controles físicos y lógicos, políticas de seguridad y planes de contingencia que permitan mantener la continuidad operativa ante incidentes o desastres.
